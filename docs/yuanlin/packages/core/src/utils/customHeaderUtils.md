# customHeaderUtils.ts

## 概述

`customHeaderUtils.ts` 是一个自定义 HTTP 请求头解析工具模块，提供了将环境变量中的字符串格式自定义请求头解析为键值对映射（`Record<string, string>`）的功能。该模块主要用于支持用户通过环境变量配置自定义 HTTP 请求头，以便在 Gemini API 请求中附加额外的头部信息（例如代理认证、追踪标识等）。

## 核心组件

### `parseCustomHeaders(envValue: string | undefined): Record<string, string>`

将字符串格式的自定义请求头解析为键值对映射。

**参数**：
- `envValue: string | undefined` — 从环境变量读取的原始字符串值。可能为 `undefined`（环境变量未设置时）。

**返回值**：
- `Record<string, string>` — 解析后的请求头键值对映射。如果输入为空或无效，则返回空对象。

**解析规则与流程**：

#### 第一步：空值处理

如果 `envValue` 为 `undefined`、空字符串或其他 falsy 值，直接返回空对象 `{}`。

#### 第二步：智能分割

使用正则表达式 `,(?=\s*[^,:]+:)` 对输入字符串进行分割。

该正则表达式的含义：
- `,` — 匹配逗号字符。
- `(?=\s*[^,:]+:)` — 正向前瞻断言（lookahead），要求逗号后面跟随：
  - `\s*` — 零个或多个空白字符。
  - `[^,:]+` — 一个或多个既不是逗号也不是冒号的字符（即下一个请求头的名称）。
  - `:` — 一个冒号（即下一个请求头名称后的分隔符）。

**这种智能分割的关键优势**：
- 它只在"逗号后面紧跟着一个新的 `key:` 模式"的位置进行分割。
- 如果请求头的值本身包含逗号或冒号（例如 `Authorization: Bearer token,extra`），该逗号不会被错误地作为分隔符，因为它后面的内容不符合 `key:` 的模式。

**示例**：
```
输入: "X-Custom:value1, Authorization:Bearer abc,def, X-Trace:123"
分割结果:
  - "X-Custom:value1"
  - "Authorization:Bearer abc,def"    ← 值中的逗号被正确保留
  - "X-Trace:123"
```

#### 第三步：逐条解析

对每个分割后的条目：
1. **去除首尾空白**：使用 `trim()` 清理条目。
2. **跳过空条目**：如果 trim 后为空字符串，跳过。
3. **定位分隔符**：使用 `indexOf(':')` 找到第一个冒号的位置。如果没有冒号，跳过该条目。
4. **提取名称和值**：
   - `name` = 冒号前的部分，trim 后的结果。
   - `value` = 冒号后的部分，trim 后的结果。
5. **验证名称**：如果 name 为空字符串，跳过该条目。
6. **存入映射**：将 `name` -> `value` 对存入结果对象。

**注意**：使用 `indexOf(':')` 而非 `split(':')` 可以正确处理值中包含冒号的情况。例如 `Authorization:Bearer abc:def` 会被解析为 `name="Authorization"`, `value="Bearer abc:def"`。

#### 相同名称的处理

如果多个条目具有相同的名称，后面的值会覆盖前面的值（因为直接使用对象属性赋值 `headers[name] = value`）。

## 关键实现细节

1. **正则表达式的精妙设计**：`/,(?=\s*[^,:]+:)/` 是该模块最核心的实现细节。通过正向前瞻断言，它实现了"仅在逗号后跟新的请求头键名时才分割"的语义，完美解决了请求头值中可能包含逗号的歧义问题。

2. **使用 `indexOf` 而非 `split` 处理冒号**：通过 `indexOf(':')` 定位第一个冒号并使用 `slice` 提取前后部分，确保了请求头值中包含冒号时的正确解析。如果使用 `split(':')`，则值中的冒号会导致错误分割。

3. **多层防御性校验**：
   - 输入为 falsy 值时直接返回空对象。
   - 空条目被跳过。
   - 无冒号的条目被跳过。
   - 空名称的条目被跳过。
   这些校验确保了函数在面对各种异常输入时都能优雅地处理，不会抛出异常。

4. **值允许为空字符串**：如果某个请求头的值部分为空（例如 `X-Empty:`），解析结果中该请求头的值为空字符串 `""`，而不是被跳过。这是合理的，因为某些 HTTP 请求头确实可以有空值。

5. **无副作用的纯函数设计**：`parseCustomHeaders` 是一个纯函数，不修改输入参数，不依赖外部状态，相同的输入始终产生相同的输出，非常适合单元测试。

6. **JSDoc 注释中的拼写小误**：源码 JSDoc 中 `"vallues"` 应为 `"values"`，这是一个轻微的拼写错误，不影响功能。
